# IoT Product MG Case

Use this file as a calibration case, not as a universal template.

This reference captures one real `MG Explainer / Information Short` pattern from the local benchmark video `1oT Product Film.mp4`.

## Why This Case Matters

This case is useful because it is:

- clearly `MG Explainer / Information Short`
- product-led rather than workflow-led
- geometry-forward and world-building-forward rather than UI-card-forward
- a strong counterweight to enterprise workflow MG examples such as `Super Rep`

Use it when the user wants:

- a product-world MG benchmark
- a geometric or concept-led explainer benchmark
- an IoT or platform-capability explanation pattern
- calibration against a reference video with stronger visual abstraction

Do not use it as proof that all MG videos should look or move this way.

## Core Case Summary

Working description:

- a product explainer for `1oT`
- focused on global IoT connectivity and device management scale
- centered on the pain of many carriers, SIMs, contracts, and operational complexity
- resolved through the `1oT Terminal`, supporting services, analytics, issue detection, future events, and API access

Approximate structure:

1. define the scale problem in the IoT business
2. introduce `1oT Terminal` as the unifying platform
3. expand into platform management and monitoring features
4. present modular value-added services
5. close on API integration and global-connectivity positioning

## Dominant Type Judgment

Primary type:

- `MG Explainer / Information Short`

Secondary supporting traits:

- `Product Promo Short`
- `Enterprise Promo`

Not primary type:

- `UI Demo / Screenflow Video`
- `Internal Promo / Internal Adoption Short`
- `Concept Narrative Short`
- `AI cinematic short film`

## Why It Is MG

Transferable MG signals in this case:

- information and product logic are the main payload
- graphic-composite logic dominates
- geometry, icon-like objects, modular scenes, and abstract spaces carry meaning
- scenes explain system value more than they depict realistic action
- sections feel product-system-led rather than performance-led
- feature blocks are staged as visual modules

## What Is Specific To This Case

These are case-specific decisions, not MG rules:

- blue, green, yellow, and violet palette
- bold geometric object language
- 2.5D or pseudo-3D spatial staging
- delivery robot and IoT fleet scenario
- telecom or carrier-management context
- `carrier services`, `issue detector`, `future events`, and API framing
- a more playful product-world tone than a typical enterprise workflow explainer

Do not generalize these details to all MG work.

## What Must Not Be Overgeneralized

Do not conclude from this case that:

- every MG piece should use geometric world-building
- every MG piece needs bright multi-color product branding
- every MG piece should stage its message as a platform adventure or product universe
- every MG piece should avoid UI-heavy sequences
- every MG piece should be about modular services and API expansion

## Reusable Guardrails

These lessons transfer well to future product-led MG projects:

- use a clear product problem-to-platform-solution arc
- let the product world itself carry explanation, not only literal screenshots
- build a coherent shape language before expanding feature modules
- make feature families feel modular and scalable
- use abstraction to simplify complexity, not to hide the logic
- let brand and product identity emerge through repeated visual rules

## Reusable Freshness Levers

These are safe places to vary future MG projects without leaving the type:

- shape language
- world-building metaphor
- feature-module architecture
- camera-space or depth illusion
- object motion grammar
- problem visualization mechanic
- service-block naming and reveal system
- closing product-positioning line

This means a future product MG can feel very different while still staying inside the MG lane.

## Step 1 Calibration

What this case teaches Step 1:

- classify a product-led modular platform explainer as `MG Explainer / Information Short` when explanation still depends mainly on designed visual modules
- allow `Product Promo Short` energy as a supporting layer without collapsing the piece into ad-only logic
- recognize that MG can be product-world-led, not only workflow-led or card-led

What Step 1 should not do:

- mistake playful world-building for narrative-short logic
- mistake stronger product presence for abandoning MG structure
- mistake modular feature sections for a pure UI demo

## Step 2 Calibration

What this case teaches Step 2:

- storyboard around product complexity, platform unification, and modular capabilities
- allow more visual world-building and object-driven transitions than a workflow-enterprise MG
- let modules and feature families become recurring scene logic
- preserve conceptual clarity while using more spatial and graphic variety

What Step 2 should not do:

- flatten this type into only card-and-label boards
- force realistic office workflow scenes where abstraction explains better
- over-narrativize the product journey into character drama

## Step 3 Calibration

What this case teaches Step 3:

- prefer `graphic-composite`
- write prompts around shape language, modular scenes, product-world objects, dashboards, feature tiles, and abstract platform spaces
- use UI or boards as support assets inside a larger product-world system
- maintain a strong visual grammar across feature families

What Step 3 should not do:

- reduce the piece to generic corporate card prompts
- overload prompts with realistic acting directions
- let every feature block use a different visual language

## Step 4 Calibration

What this case teaches Step 4:

- execute in this order: shape language -> world modules -> product hero assets -> feature blocks -> support UI panels -> final composites
- keep recurring geometry, product objects, and service blocks reusable
- validate visual grammar before scaling feature variants
- separate world-building assets from literal informational overlays

What Step 4 should not do:

- start from literal UI output before the product-world system is defined
- treat the project like an enterprise workflow MG with only card families
- mix hero world assets and utility data modules without structure

## Recommended Use Rule

If a future project feels adjacent to this case:

- transfer the MG guardrails
- transfer the product-led modular structure if it fits the problem
- choose fresh palette, shape language, world metaphor, and module system for the new job

If a future project is more workflow-led or people-led:

- do not force this geometric world-building pattern onto it
- use it only to remember that MG has multiple valid sub-lanes
